# Driver installation strategy

This project must preserve the Windows USB Audio interface and install WinUSB
only on the Philips control interface.

Current validated target:

```text
USB\VID_0471&PID_0111&MI_03
```

Known audio interface that must not be changed:

```text
USB\VID_0471&PID_0111&MI_00
```

## Why only MI_03

The Philips USB PC Link units expose a composite USB device. On the validated
FWM589/MCM590 style hardware, `MI_00` is the USB Audio render interface used by
Windows for playback. The PC Link enable command is sent through vendor control
transfers, and the radio button endpoint is exposed on the control/HID side.

Replacing the whole composite device, or replacing `MI_00`, breaks normal audio
playback. The commercial installer must therefore bind WinUSB only to the
interface-specific hardware ID for `MI_03`.

## Installation flow

The shipping installer should use this sequence:

1. Detect connected Philips USB PC Link devices.
2. Match the device against a supported profile.
3. Confirm that the audio interface is still present and not using WinUSB.
4. Validate the bundled INF before installing it.
5. Install the driver package with administrator rights.
6. Rescan devices.
7. Confirm that `MI_03` uses `WinUSB` and that `MI_00` is still present.
8. Send the PC Link enable packets after install, after reconnect, and when the
   user switches the stereo back to PC Link mode.

For the current package, the only supported install target is:

```text
USB\VID_0471&PID_0111&MI_03
```

Do not use a generic Philips VID/PID INF. Do not use a whole-device WinUSB INF.
Do not install WinUSB over `USB\VID_0471&PID_0111&MI_00`.

## Driver package validation

Before calling `pnputil`, the installer must verify:

- The INF exists in `driver/winusb-philips-mi03`.
- The INF `DeviceID` is exactly `VID_0471&PID_0111&MI_03`.
- The install section binds `USB\%DeviceID%`, not a broader match.
- The INF text does not define `VID_0471&PID_0111&MI_00`.
- The catalog file exists.

The helper script `scripts/install-winusb-mi03.ps1` performs these checks.

## Signing policy

The current local package was generated by libwdi/Zadig and uses a local
self-signed certificate. That is acceptable for development and recovery work,
but it is not the right default for a commercial or public release.

Recommended release policy:

- Sign the application executable with a trusted code-signing certificate.
- Ship a production-signed driver package whenever possible.
- Use Microsoft driver-store installation tools rather than custom registry
  writes.
- If a community/test build needs a self-signed libwdi package, show an explicit
  consent screen before trusting the local certificate.

The script supports test packages through `-AcceptLocalTestCertificate`. A
production installer should not need that switch.

## Restore flow

Restoration should remove only the WinUSB package currently assigned to `MI_03`.
The helper script `scripts/restore-mi03-hid.ps1`:

1. Finds the connected `USB\VID_0471&PID_0111&MI_03` device node.
2. Verifies that it is currently using WinUSB.
3. Reads the published INF name assigned to that node.
4. Deletes that driver package with `pnputil /delete-driver ... /uninstall`.
5. Rescans devices so Windows can bind the original HID/input driver again.
6. Confirms that `MI_00` was not targeted.

If Windows keeps the previous binding cached, ask the user to unplug/replug the
USB cable and run the verification script again.

## Multi-model support

Do not try to support every Philips USB PC Link model with one wildcard driver.
Use explicit profiles. Each profile should include:

- USB VID/PID.
- Audio interface number.
- Control interface number.
- Expected control endpoint, if any.
- Activation packet sequence.
- Button report format.
- Verified models and firmware notes.

Suggested future manifest:

```json
{
  "profiles": [
    {
      "id": "philips-0471-0111-mi03",
      "models": ["FWM589", "MCM590", "MCM530"],
      "vid": "0471",
      "pid": "0111",
      "audio_interface": "MI_00",
      "control_interface": "MI_03",
      "driver": "driver/winusb-philips-mi03",
      "activation_packets": [
        {"request_type": "0x40", "request": "0x04", "value": "0x0000", "index": "0xA4EF", "data": "01"},
        {"request_type": "0x40", "request": "0x04", "value": "0x0000", "index": "0xA4F0", "data": "FF"}
      ]
    }
  ]
}
```

New models should be added only after a descriptor capture and a real playback
test. The installer should refuse unknown profiles rather than guessing.

## QA checklist

Before publishing a release:

- Fresh Windows 10 install test.
- Fresh Windows 11 install test.
- Install with the stereo already connected.
- Install before the stereo is connected.
- Reconnect USB while the app is running.
- Switch away from PC Link and back again.
- Confirm that Windows volume changes the Philips endpoint.
- Confirm that radio volume still changes the amplifier volume.
- Confirm Play/Pause, Stop, Next, Previous, Mute, Volume Up, and Volume Down.
- Run restore and confirm Windows returns `MI_03` to HID/input.
- Confirm the audio interface remains available throughout install and restore.
